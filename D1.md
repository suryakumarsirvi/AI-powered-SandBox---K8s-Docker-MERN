A Kubernetes Service Account is a non-human identity resource that provides a distinct identity for processes running inside Pods, enabling them to authenticate with the Kubernetes API server or external services.  Unlike user accounts intended for humans, service accounts are managed within the cluster and are essential for implementing Role-Based Access Control (RBAC) to enforce the principle of least privilege.

To apply a service account to a Kubernetes deployment, you must first create the service account and bind it to the necessary permissions using Roles and RoleBindings. Then, you specify the service account name in the pod template of your deployment YAML file.

In Kubernetes, a RoleBinding grants the permissions defined in a Role to a specific ServiceAccount within a namespace.  This is the standard mechanism for authorization, ensuring that pods running under that service account have the necessary access to API resources.

Relationship-Based Access Control (ReBAC) is an authorization model that grants permissions based on the specific relationships between users and individual resource instances, such as files.  Unlike traditional Role-Based Access Control (RBAC), which assigns permissions to users globally, ReBAC defines roles at the resource level (e.g., Folder#Owner or File#Editor), allowing for granular, context-aware access.
